# Circle Siege 基于 23 种设计模式的需求文档

## 1. 文档说明

- 项目名称：`Circle Siege`
- 项目类型：基于 `pygame` 的 2D 竞技/生存类枪战游戏
- 文档目的：用 `GoF 23 种设计模式` 组织项目需求与架构约束
- 适用范围：当前单机原型、后续竞技化版本、模式扩展版本
- 基线日期：`2026-06-23`

## 2. 说明与原则

先明确一件事：`需求文档` 和 `设计模式` 不是同一个层次。

- 需求文档回答的是“系统要做什么”
- 设计模式回答的是“系统应该怎么组织代码和对象关系”

因此本文件采用折中方式：

1. 先给出项目功能需求与非功能需求。
2. 再把这些需求映射到 `23 种设计模式`。
3. 明确哪些模式当前必须落地，哪些只作为扩展预留。

目标不是“为了凑 23 种模式而过度设计”，而是用模式把项目做得更清晰、更可维护。

## 3. 项目总体需求

## 3.1 总体目标

本项目需要构建一款具有以下特征的 2D 枪战游戏：

- 以 `pygame` 为核心框架
- 支持单机玩家与 `Bot` 对战
- 具备缩圈生存玩法
- 支持多张主题地图
- 支持武器、补给、手雷、治疗、结算、HUD
- 具备后续扩展团队竞技、占点模式和联网版本的结构基础

## 3.2 功能需求

### 3.2.1 战斗基础需求

- 玩家必须支持移动、瞄准、射击、换弹、切枪、治疗、投掷手雷。
- 敌方 `Bot` 必须支持巡逻、搜敌、追击、撤退、进圈、拾取和射击。
- 游戏必须支持伤害计算、护甲计算、死亡判定和结算统计。

### 3.2.2 武器与道具需求

- 系统必须支持多种武器类型，如冲锋枪、卡宾枪、霰弹枪、精确射手步枪。
- 每种武器必须支持独立参数：伤害、射速、弹夹、散射、射程、弹药类型。
- 系统必须支持弹药、医疗包、武器掉落、补给箱。

### 3.2.3 地图与场景需求

- 游戏必须支持多主题地图，如海港工业区、荒漠前哨站、雨林遗迹。
- 地图必须支持出生点、掩体、补给刷新点和安全区逻辑。
- 场景必须支持主菜单、战斗场景、结算场景、设置场景。

### 3.2.4 规则需求

- 当前必须支持生存模式。
- 规则系统必须支持安全区缩圈、毒圈伤害、补给投放和胜负判定。
- 架构必须预留团队竞技、占点模式、复活模式的扩展接口。

### 3.2.5 表现与体验需求

- HUD 必须显示血量、护甲、弹药、武器、公告、击杀播报、缩圈状态。
- 系统必须支持粒子、爆炸、伤害数字、镜头跟随。
- 后续应支持更强的炫酷表现，如局部光效、震屏、视差背景、低血量边缘提示。

## 3.3 非功能需求

- 目标帧率应稳定在 `60 FPS` 左右。
- 代码应便于扩展新武器、新地图、新模式。
- 模块职责必须清晰，避免巨型类失控。
- 配置数据应集中管理，尽量减少硬编码。
- 调试能力必须保留，便于测试玩法、AI 和缩圈节奏。

## 4. 23 种设计模式总体映射

| 创建型 | 抽象工厂 `Abstract Factory` | 地图主题资源包、UI 主题、敌人阵营套件 | P1 |

| 创建型 | 原型 `Prototype` | 武器模板、敌人模板、粒子模板 | P1 |
| 结构型 | 适配器 `Adapter` | Tiled 地图、外部资源、未来联网层接口适配 | P1 |
| 结构型 | 桥接 `Bridge` | 武器逻辑与表现分离、角色控制与渲染分离 | P1 |

| 结构型 | 外观 `Facade` | 战斗系统统一入口、资源系统统一入口 | P0 |
| 结构型 | 享元 `Flyweight` | 粒子、贴图、字体、重复地图元素共享 | P0 |
| 结构型 | 代理 `Proxy` | 延迟加载资源、远端玩家代理、调试代理 | P2 |
| 行为型 | 责任链 `Chain of Responsibility` | 输入处理、伤害结算、命中修正、UI 事件流 | P1 |
| 行为型 | 命令 `Command` | 输入映射、调试快捷键、可重放操作 | P0 |
| 行为型 | 解释器 `Interpreter` | 规则表达式、关卡条件、脚本化触发器 | P2 |
| 行为型 | 迭代器 `Iterator` | 角色集合、子弹集合、UI 元素遍历 | P2 |
| 行为型 | 中介者 `Mediator` | `BattleScene` / `MatchManager` 协调对象通信 | P0 |
| 行为型 | 备忘录 `Memento` | 设置存档、局内快照、结算快照 | P1 |
| 行为型 | 观察者 `Observer` | HUD、公告、击杀播报、音效响应 | P0 |
| 行为型 | 状态 `State` | 玩家状态、Bot 状态、场景状态、对局状态 | P0 |
| 行为型 | 策略 `Strategy` | AI 策略、武器开火策略、模式规则策略 | P0 |
| 行为型 | 模板方法 `Template Method` | 场景生命周期、角色更新流程、UI 刷新流程 | P0 |
| 行为型 | 访问者 `Visitor` | 调试统计、导出报表、对象检查器 | P2 |

优先级说明：

- `P0`：当前项目应尽快落地
- `P1`：中期适合落地
- `P2`：先保留接口，不急于实现

## 5. 按 23 种设计模式展开的需求

## 5.1 创建型模式需求

### 5.1.1 单例模式 `Singleton`

需求：

- 系统必须提供唯一配置中心，统一管理屏幕、音量、帧率和数值配置。
- 系统必须提供统一资源管理入口，避免贴图、字体、音效重复加载。
- 系统必须提供统一音频管理入口，避免多个模块直接控制 `pygame.mixer`。

建议对象：

- `GameConfig`
- `ResourceManager`
- `AudioManager`
- `EventBus`

验收标准：

- 任意模块获取到的是同一份配置和资源上下文。
- 资源不会因重复实例化而浪费内存。

### 5.1.2 工厂方法模式 `Factory Method`

需求：

- 场景对象必须通过工厂创建，而不是在各处散乱 `new`。
- 子弹、手雷、补给物、Bot、武器掉落必须通过统一创建逻辑生成。

建议对象：

- `SceneFactory`
- `ProjectileFactory`
- `PickupFactory`
- `BotFactory`

验收标准：

- 新增一个子弹类型或场景时，不需要修改大量调用方代码。

### 5.1.3 抽象工厂模式 `Abstract Factory`

需求：

- 不同地图主题应能成套切换地表、障碍、配色、UI 强调色和环境音。
- 不同阵营角色应能成套切换角色皮肤、名字风格、武器池和颜色方案。

建议对象：

- `ThemeFactory`
- `FactionFactory`
- `UISkinFactory`

验收标准：

- 切换地图主题时，系统能一次性切换整套资源和样式。

### 5.1.4 建造者模式 `Builder`

需求：

- 对局初始化必须按步骤构建：地图、玩家、Bot、补给、安全区、HUD、规则。
- 结算页应根据战斗结果构建摘要对象。

建议对象：

- `MatchBuilder`
- `MapBuilder`
- `SummaryBuilder`

验收标准：

- 对局构建过程清晰，便于后续插入新步骤。

### 5.1.5 原型模式 `Prototype`

需求：

- 武器、粒子、Bot 行为参数应支持模板复制，而不是每次从零组装。
- 常见拾取物和爆炸效果应可从原型快速克隆。

建议对象：

- `WeaponSpec`
- `ParticleTemplate`
- `BotTemplate`

验收标准：

- 新增一把枪时，可从现有枪模板克隆并微调。

## 5.2 结构型模式需求

### 5.2.1 适配器模式 `Adapter`

需求：

- 系统应支持将外部地图工具、资源格式或未来网络同步数据适配到内部统一接口。

建议对象：

- `TiledMapAdapter`
- `ExternalAssetAdapter`
- `RemoteStateAdapter`

验收标准：

- 外部格式变化不会直接污染核心战斗逻辑。

### 5.2.2 桥接模式 `Bridge`

需求：

- 角色控制逻辑与角色渲染逻辑必须分离。
- 武器射击逻辑与武器视觉表现必须分离。

建议对象：

- `CharacterController` + `CharacterRenderer`
- `WeaponLogic` + `WeaponView`

验收标准：

- 替换角色贴图或武器表现时，不应影响核心射击逻辑。

### 5.2.3 组合模式 `Composite`

需求：

- HUD 必须支持面板、子面板、控件的树形结构。
- 地图障碍群、装饰群、掉落群应支持分层组合。

建议对象：

- `UIComponent`
- `UIPanel`
- `UILeaf`
- `MapNode`

验收标准：

- UI 和地图对象可统一遍历、统一更新、统一绘制。

### 5.2.4 装饰器模式 `Decorator`

需求：

- 角色可能获得临时效果，如加速、护盾、灼烧、减速。
- 武器可能附加特殊效果，如爆炸弹、穿透弹、毒性弹。

建议对象：

- `CharacterEffectDecorator`
- `WeaponModifierDecorator`

验收标准：

- 新增一个临时效果时，不需要修改角色本体类。

### 5.2.5 外观模式 `Facade`

需求：

- 战斗场景需要一个统一接口来驱动战斗逻辑、碰撞、粒子、音效和结算。
- 资源系统需要一个统一接口屏蔽底层贴图、字体、音频细节。

建议对象：

- `BattleFacade`
- `AssetFacade`

验收标准：

- 上层调用方只需面向一个入口，而不需要知道多个底层对象的细节。

### 5.2.6 享元模式 `Flyweight`

需求：

- 相同粒子、字体、贴图、地图砖块、弹壳和 UI 图标应共享数据。
- 高重复对象应减少重复内存分配。

建议对象：

- `ParticleFlyweight`
- `TileFlyweight`
- `IconFlyweight`

验收标准：

- 大量重复对象出现时，内存和性能开销可控。

### 5.2.7 代理模式 `Proxy`

需求：

- 资源加载应支持延迟代理。
- 后续联机版本中，远端玩家状态可通过代理对象访问。
- 调试模块可为某些对象提供监视代理。

建议对象：

- `LazyAssetProxy`
- `RemotePlayerProxy`
- `DebugProxy`

验收标准：

- 重资源只有在真正需要时才加载。

## 5.3 行为型模式需求

### 5.3.1 责任链模式 `Chain of Responsibility`

需求：

- 输入事件应经过多个处理环节，例如菜单处理、战斗处理、调试处理。
- 伤害结算应支持链式修正，如护甲减伤、爆头加成、毒圈伤害、护盾吸收。

建议对象：

- `InputHandlerChain`
- `DamageHandlerChain`

验收标准：

- 新增一种命中修正时，不需要重写整条伤害逻辑。

### 5.3.2 命令模式 `Command`

需求：

- 键盘、鼠标、调试快捷键应映射为命令对象，而不是硬编码在大量 `if` 分支里。
- 后续若要支持操作重放、录制或自定义按键，必须依赖命令模型。

建议对象：

- `MoveCommand`
- `FireCommand`
- `ReloadCommand`
- `SpawnSupplyCommand`

验收标准：

- 更换键位映射时，不应破坏战斗逻辑。

### 5.3.3 解释器模式 `Interpreter`

需求：

- 后续规则脚本、触发器表达式、教学关卡条件应支持解释执行。

建议对象：

- `RuleExpressionInterpreter`
- `TriggerInterpreter`

验收标准：

- 可用较简洁的规则表达式定义触发条件。

### 5.3.4 迭代器模式 `Iterator`

需求：

- 角色、子弹、拾取物、UI 节点、公告列表应支持统一遍历。

建议对象：

- `CharacterIterator`
- `ProjectileIterator`
- `UIIterator`

验收标准：

- 遍历方式统一，不暴露底层容器细节。

### 5.3.5 中介者模式 `Mediator`

需求：

- 玩家、Bot、地图、碰撞、粒子、音效和 HUD 之间不应彼此直接强耦合。
- 必须由一个中介对象统筹协调战斗内消息流。

建议对象：

- `MatchManager`
- `BattleSceneMediator`

验收标准：

- 新增一个系统时，优先接入中介者，而不是让所有对象彼此引用。

### 5.3.6 备忘录模式 `Memento`

需求：

- 设置界面参数必须能保存与恢复。
- 结算快照必须可保留本局关键数据。
- 调试模式下可记录局内快照，便于复现问题。

建议对象：

- `SettingsMemento`
- `MatchSnapshot`
- `DebugSnapshot`

验收标准：

- 用户可以恢复设置，开发者可以回看关键局面。

### 5.3.7 观察者模式 `Observer`

需求：

- 玩家生命值变化后，HUD 必须自动刷新。
- 击杀事件发生后，公告、音效、粒子、统计模块必须自动响应。

建议对象：

- `EventBus`
- `HUDObserver`
- `KillfeedObserver`
- `AudioObserver`

验收标准：

- 新增一个监听模块时，不需要侵入核心战斗类。

### 5.3.8 状态模式 `State`

需求：

- 玩家状态必须区分待机、移动、射击、换弹、治疗、受击、死亡。
- Bot 状态必须区分巡逻、搜索、追击、交战、撤退、进圈。
- 场景状态必须区分菜单、战斗、结算、暂停。

建议对象：

- `PlayerState`
- `BotState`
- `SceneState`
- `MatchState`

验收标准：

- 复杂状态切换不应靠大量布尔变量硬撑。

### 5.3.9 策略模式 `Strategy`

需求：

- 不同枪械应有不同开火策略，如单发、连发、霰弹散射。
- 不同 Bot 应支持不同 AI 策略，如激进、保守、搜刮优先。
- 不同对局模式应支持不同胜负策略。

建议对象：

- `FireStrategy`
- `AIStrategy`
- `WinConditionStrategy`

验收标准：

- 更换行为时，应替换策略对象，而不是改动主类。

### 5.3.10 模板方法模式 `Template Method`

需求：

- 所有场景都应遵循统一生命周期：处理输入、更新逻辑、绘制界面。
- 所有角色都应遵循统一更新模板：输入/AI 决策、移动、碰撞、攻击、状态结算。

建议对象：

- `BaseScene`
- `CharacterBase`
- `HUDBase`

验收标准：

- 子类只重写差异部分，不重复编写通用流程。

### 5.3.11 访问者模式 `Visitor`

需求：

- 开发与调试工具需要对角色、武器、地图对象做统一检查和统计。
- 后续导出战绩报表或地图分析时，适合通过访问者收集数据。

建议对象：

- `DebugVisitor`
- `StatsVisitor`
- `ExportVisitor`

验收标准：

- 新增统计功能时，不需要频繁修改实体类本身。

- `Bot`：通过 AI 形成战斗压力。
- `开发/测试人员`：通过调试面板观察行为、平衡和性能。
- `后续联机玩家`：预留角色，不在当前首版强制实现。

## 6.2 核心对象

- `Projectile`
- `Grenade`
- `Pickup`
- `Map`
- `HUDRenderer`
- `Camera`
- `ParticleSystem`

## 7. 模式落地优先级建议

## 7.1 当前必须落地的模式

这些模式最值得优先实现：

- `Singleton`
- `Factory Method`
- `Builder`
- `Composite`
- `Facade`
- `Flyweight`
- `Command`
- `Mediator`
- `Observer`
- `State`
- `Strategy`
- `Template Method`

原因：

- 这些模式直接决定当前项目会不会失控。
- 它们与 `pygame` 项目最匹配。
- 它们能最快改善结构、扩展性和维护成本。

## 7.2 中期适合落地的模式

- `Abstract Factory`
- `Prototype`
- `Adapter`
- `Bridge`
- `Decorator`
- `Chain of Responsibility`
- `Memento`

原因：

- 这些模式适合在地图、美术、规则和状态效果变复杂之后再引入。

## 7.3 后期预留即可的模式

- `Proxy`
- `Interpreter`
- `Iterator`
- `Visitor`

原因：

- 这些模式更多服务大型化、工具化或联网扩展。
- 当前强行引入，收益不高。

## 8. 验收标准

### 8.1 结构验收

- 不允许所有逻辑集中到一个超大类中。
- 新增武器、地图、模式时，改动范围应局限在对应模块。
- 资源、配置、事件和音频入口必须统一。

### 8.2 功能验收

- 玩家可完整打完一局生存模式。
- Bot 能稳定参与战斗与进圈。
- HUD、公告、结算和调试信息能正确反映战局。

### 8.3 模式验收

- 当前落地的 `P0` 模式应能在代码结构中明确对应。
- 模式使用要为需求服务，不允许无意义堆砌。

## 9. 开发建议

建议按以下顺序推进：

1. 先用 `Template Method + State + Strategy + Command` 稳定角色与场景流程。
2. 再用 `Factory Method + Builder + Mediator + Observer` 稳定对象创建与战斗协作。
3. 然后用 `Composite + Facade + Flyweight` 稳定 UI、资源和性能。
4. 最后根据复杂度再引入 `Decorator / Abstract Factory / Adapter / Memento`。

## 10. 结论

如果直接把这款 `pygame` 游戏当成普通脚本堆功能，后面一定会出现这些问题：

- 类之间相互引用混乱
- 新增地图和模式成本过高
- 角色状态、武器逻辑和 UI 逻辑纠缠
- 调试、平衡和扩展越来越困难

按本文件的方式，把需求和 `23 种设计模式` 对齐后，项目会更容易形成稳定工程结构：

- 需求边界清晰
- 模块职责清晰
- 代码扩展路径清晰
- 后续做炫酷场景、团队模式、联机扩展时也更有基础
